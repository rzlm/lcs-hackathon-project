### Layer-by-Layer Explanation

#### Layer 1: External Data Sources

**Toronto Open Data** is the primary source of truth for service existence and shelter occupancy. The key datasets are:
- **Daily Shelter & Overnight Service Occupancy & Capacity** — updated hourly. Contains bed counts, occupancy rates, addresses, and program type per shelter.
- **Warming Centres** — seasonal (active November–April roughly). Updated when city activates/deactivates.
- **Cooling Centres** — seasonal (active June–September). Same structure.

These are public REST APIs returning JSON. No API key required. Reliability is generally good but outages happen — the system must gracefully degrade.

**Weather API (Open-Meteo)** is free, no-key-required, and returns hourly forecasts. We use it to correlate extreme weather with predicted shelter demand (below -10°C or above 35°C = high demand heuristic).

**Crowdsourced data** comes from in-app user reviews. A user taps "Was space available when you arrived?" — yes/no + optional text. These feed back through the same Supabase pipeline and are weighted with a trust score.

#### Layer 2: Supabase Backend

Supabase acts as the single backend service. It provides:
- **PostgreSQL** — relational storage for all normalized service data
- **Edge Functions** (Deno runtime) — scheduled cron ingestion, SMS webhook handler, review moderation
- **PostgREST** — auto-generated REST API from table schema
- **Realtime** — optional WebSocket channel for live occupancy updates (V2)
- **Storage** — not needed for MVP

The backend runs the data pipeline: raw Toronto Open Data JSON → normalized service records → availability_reports with confidence scoring → served to clients.

#### Layer 3: Device Storage (SQLite / IndexedDB)

On mobile, `expo-sqlite` provides a full SQLite database embedded in the app. This is the offline-first foundation. All data consumed by the UI is read from SQLite first. Network calls only update this local store.

On web, we use a combination of `localStorage` (for small, critical data like recent searches) and `IndexedDB` (for larger service data). The abstraction layer in `src/services/db/` handles the difference transparently.

**Why SQLite over AsyncStorage?** AsyncStorage is key-value only and caps out quickly. SQLite supports queries — we need `WHERE type = 'shelter' AND wheelchair_accessible = 1 ORDER BY distance ASC`. There is no good alternative for that on device.

#### Layer 4: UI Layer

The React Native / Expo UI reads exclusively from local storage and dispatches sync events. It never calls Supabase directly from a screen component. All data access goes through `src/hooks/` which pull from the local DB.

Mobile uses `react-native-maps` (MapKit on iOS, Google Maps on Android). Web uses Leaflet (rationale in Section E). Platform divergence is handled via `.web.tsx` file variants — consistent with the existing pattern in this codebase.

## B. Core Features

### 1. Interactive Map with Availability Color-Coding

**User-facing behavior:** The map opens centered on the user's current location. Service markers appear within a configurable radius (default 5km). Each marker is color-coded:

| Color | Meaning | Condition |
|-------|---------|-----------|
| Green | Available — space likely open | occupancy < 80% OR recent positive review |
| Yellow | Limited — filling up | occupancy 80–95% OR mixed reviews |
| Red | Full / Unknown | occupancy > 95%, no data, or closed |
| Gray | Permanently closed / Seasonal (off-season) | |

**Implementation details:**

The `availability_score` (0.0–1.0) is computed server-side and stored on the `services` table. The UI maps this to a color:

```typescript
// src/utils/scoring.ts
export function scoreToColor(score: number | null): MarkerColor {
  if (score === null) return 'gray';
  if (score >= 0.5) return 'green';
  if (score >= 0.2) return 'yellow';
  return 'red';
}
```

**Marker clustering:** On mobile, `react-native-maps` does not have native clustering. We implement a lightweight client-side cluster using a grid-based approach (divide viewport into cells, group nearby markers). On web, Leaflet's `leaflet.markercluster` plugin handles this. See Section E for platform map abstraction.

The cluster marker shows a count badge and uses the color of the most urgent item in the cluster (if any shelter is red, the cluster is red). This helps users immediately spot critical availability problems even when zoomed out.

**Tap behavior:** Tapping a marker opens a bottom sheet (using `react-native-reanimated` which is already installed). The sheet shows: name, address, phone, hours, current availability, last-updated timestamp, and quick action buttons (call, directions, review).

### 3. User Reviews

**Design philosophy:** Reviews must be frictionless. No account, no login. The simpler the better. Someone who just left a shelter and wants to help the next person should be able to do it in 5 seconds.

**Review flow:**

```
Service Detail Screen
        │
        ▼
[ Was space available when you arrived? ]
        │
   ┌────┴────┐
   │         │
[YES ✓]   [NO ✗]
   │         │
   └────┬────┘
        │
[ Optional: add a note (max 140 chars) ]
[ __________________________________ ]
[ Skip ]          [ Submit ]
        │
        ▼
Saved to local SQLite reviews table (is_synced = false)
Optimistically shown in UI immediately
        │
        ▼ (when online)
Synced to Supabase user_reviews table
```

**Data model for a review:**

```
service_id     — which service
was_available  — boolean (the core signal)
comment_text   — optional, max 140 chars
created_at     — timestamp (local time, synced later)
is_synced      — false until confirmed by server
```

**Aggregated display:** Each service shows a "community reports" summary: "7 of 9 people found space in the last 24 hours" — derived from `was_available` counts in the last 24h window. No individual reviews shown unless comment_text is non-null and passes basic profanity filter.

**Staleness:** Reviews older than 48 hours are visually dimmed. The system treats them as lower-confidence signals when computing `availability_score`.

### 4. Offline-First Behavior

**Mental model:** The app behaves as if it is always offline and occasionally gets a chance to sync. Network is treated as a bonus, not a requirement.

**What is always available offline:**

| Data | Stored In | Cache Size (est.) | Eviction |
|------|-----------|-------------------|----------|
| Service listings (all Toronto) | SQLite `services` | ~500KB for ~200 services | Never (until app update) |
| Contact info, hours, address | SQLite `services` | Included above | Never |
| Last-known availability | SQLite `availability` | ~100KB | Keep last 7 days, then evict |
| User's own reviews (unsynced) | SQLite `reviews` | ~50KB | Never until synced |
| Map tiles (downtown Toronto core) | SQLite `map_tiles` | ~8–12MB for zoom 10–15 | LRU, 30-day expiry |
| User preferences / recent searches | AsyncStorage | ~5KB | Never |

**What is stale-ok (not critical to cache aggressively):**

| Data | Strategy |
|------|----------|
| Real-time occupancy numbers | Show last known with age indicator ("Updated 3h ago") |
| Weather data | Show last known or omit |
| Remote reviews from other users | Show last cached batch |
| News / alerts | Show last cached or hide if stale > 24h |

**Cache sizing rationale:** A user on a $30 prepaid phone with 16GB storage should not feel the app is eating space. Total target: under 20MB including code. Map tiles are the main variable — we pre-cache only the downtown Toronto core at zoom levels 10–15, which covers 95% of service use cases. Users outside that area get blank tiles with service markers still shown (markers are always displayed, tiles are cosmetic).

**Eviction policy for availability data:**
- Keep all records for services the user has recently viewed (last 7 days of views)
- Keep all records < 7 days old
- Delete records > 7 days old on app startup (after sync completes)
- Map tiles: LRU (least recently used), max 500 tiles, no tile older than 30 days

### 5. Sync Strategy

**Sync triggers (in priority order):**

1. **App foreground** — when the app comes to the foreground after being in background > 15 minutes, trigger a full sync
2. **Manual pull-to-refresh** — user explicitly requests fresh data on the map or list screens
3. **Background sync** — if `expo-background-fetch` is permitted (iOS: limited; Android: more flexible). Best-effort, not guaranteed.
4. **On review submission** — immediately attempt to sync the user's new review

**Sync algorithm:**

```
function syncWithServer():
  1. Read last_sync_at from sync_log
  2. GET /sync?since={last_sync_at} from Supabase
     → Returns: {services: [...], availability: [...], reviews: [...]}
  3. Upsert services into local SQLite (by service_id)
  4. Insert new availability records
  5. Insert new reviews (remote crowdsourced)
  6. Push unsynced local reviews to POST /reviews
  7. Write new sync_log entry (synced_at = now, records_updated = count, success = true)
  8. Notify UI via Zustand store update
```

**Conflict resolution:**

The primary conflict scenario is: user submits a review while offline, then syncs when reconnected, but the server already has a newer availability report for the same service.

Resolution rules:
- **Availability data:** Server always wins. Server-side data from Toronto Open Data is more authoritative than client-side inference.
- **User reviews:** Client always wins for their own reviews (they're append-only, not editable). If a duplicate review is detected (same user fingerprint + service_id + within 1 hour), the server deduplicates keeping the later timestamp.
- **Service metadata (hours, address):** Server always wins. The client never writes to this.

**User fingerprinting (no account):** We generate a stable UUID on first app install, store it in SecureStore, and send it as an anonymous identifier with reviews. This is never linked to any PII. It exists solely to:
1. Prevent spam (rate limit reviews per fingerprint)
2. Allow deduplication
3. Allow a user to see their own past reviews

**Network failure handling:**
- If sync fails, show a non-intrusive banner: "Showing data from [time ago]. Tap to retry."
- Reviews queued in SQLite with `is_synced = false` — they will sync on next successful connection
- Never show an error dialog unless the user explicitly requested a refresh and it failed

### 6. SMS Fallback

See Section G for the full SMS interaction flow.

**Why SMS matters for this population:** A significant subset of people experiencing homelessness use prepaid phones with limited data but unlimited texting. SMS works when data doesn't. It works on any phone, even a flip phone. It requires no app install. It is the lowest-barrier channel we can offer.

### 7. Crowdsourced Updates (V2)

**Trust scoring system (V2 only — too complex for hackathon MVP):**

Each review contributor has a `trust_score` (0.0–1.0). Initially 0.5 (neutral). Score adjusts based on:
- Accuracy correlation: when the user's `was_available = true` is followed by other users confirming the same, score increases
- Consistency: reviews that frequently contradict official data without explanation lower the score
- Frequency: submitting too many reviews in a short window (bot-like behavior) flags for review

Reviews from high-trust contributors (>0.7) are weighted more heavily in `availability_score` computation.

**Moderation (V2):**
- Free-text comments run through a basic profanity filter (client-side word list, ~2KB)
- Flagging: users can long-press any comment and "report inappropriate"
- Supabase RLS: flagged reviews are hidden from the public feed after 3 reports, pending manual review

## C. Data Layer

### Handling Non-Real-Time Data

Toronto Open Data updates shelter occupancy approximately every hour during active hours, sometimes less frequently overnight. The data has a non-trivial lag — the city collects from shelters' intake systems, not live counts.

**Strategy: "Last known + confidence age"**

Every piece of availability data carries:
1. `reported_at` — when the source data was valid
2. `confidence_score` — 0.0–1.0, decays over time

```
confidence_score = base_score × time_decay_factor × source_reliability_factor

where:
  time_decay_factor = e^(-0.1 × hours_since_reported)
  // halves roughly every 7 hours

  source_reliability_factor:
    toronto_open_data = 0.95
    user_review       = 0.65
    predicted_only    = 0.40
```

When showing data to users, we show this visually: "Last updated 2h ago (estimated)" with a small clock icon. This sets honest expectations without being alarming.

### Availability Heuristics

When real-time data is unavailable or stale, we estimate using historical patterns:

**Heuristic 1: Time-of-day pattern**

Shelters typically fill up between 4–9 PM. By 10 PM, most beds are assigned. Early morning (6–10 AM) after sign-out, availability is higher.

```
hour  | typical_occupancy_modifier
------|--------------------------
0–6   | +0.05 (some turnover from overnight)
6–10  | +0.20 (morning departures)
10–16 | baseline
16–21 | -0.20 (evening surge)
21–24 | -0.10 (settled for the night)
```

**Heuristic 2: Weather correlation**

Extreme cold (< -10°C) or extreme heat (> 33°C) increases demand by historically 15–30%. Warming/cooling centres open, shelters fill faster.

```typescript
function weatherDemandModifier(tempC: number): number {
  if (tempC < -10) return 0.30;  // High surge
  if (tempC < 0)   return 0.15;  // Moderate surge
  if (tempC > 33)  return 0.20;  // Heat wave surge
  if (tempC > 28)  return 0.10;  // Warm weather surge
  return 0;
}
```

**Heuristic 3: Day-of-week**

Fridays and Saturdays tend to have slightly higher turnover. Mondays see more intakes after weekend displacement. These are weak signals but worth including as tie-breakers.

**Heuristic 4: Seasonal trend**

November–March: shelters consistently at 90–100% occupancy city-wide. April–October: more capacity available. This sets a baseline floor/ceiling for predictions.

**Lightweight scoring (MVP):**

```
availability_score = base_occupancy_rate
  + time_of_day_modifier
  - weather_demand_modifier
  + recent_positive_reviews_bonus  (0–0.15 based on last 6h)
  - recent_negative_reviews_penalty (0–0.15)

clamped to [0.0, 1.0]
```

No machine learning needed for MVP. These rule-based adjustments are explainable, debuggable, and sufficient for demo. In V2, a simple logistic regression on historical occupancy data would improve accuracy.

### Tile Caching Strategy

Map tiles are raster images (256×256 px PNG) fetched by the map library. For offline support, we intercept tile requests and serve from SQLite when the URL has a cached entry.

**Pre-caching on first sync:**

Define a Toronto downtown bounding box:
```
SW: 43.6200° N, -79.4500° W
NE: 43.7200° N, -79.3200° W
```

For zoom levels 10–15, this covers approximately 400–600 tiles. At ~15KB average per tile, that's ~7–9MB. Acceptable.

We do NOT pre-cache zoom levels 16–18 (street level) — too many tiles. These are fetched on-demand when the user has data, and cached opportunistically.

**Tile request interceptor:**

```typescript
// On mobile: custom URLProtocol or fetch override
// On web: Service Worker intercepts tile requests
async function resolveTile(url: string): Promise<Blob | null> {
  const cached = await db.getMapTile(url);
  if (cached && !isTileExpired(cached.cached_at)) {
    return cached.data_blob;
  }
  if (!isOnline()) {
    return cached?.data_blob ?? null; // Serve stale rather than nothing
  }
  const fresh = await fetch(url);
  await db.cacheMapTile(url, await fresh.blob());
  return fresh.blob();
}
```

**On web:** Service Workers handle tile caching via the Cache API. This is more powerful than SQLite blobs for the web platform. The `src/services/db/web.ts` implementation uses the Cache API for tiles and IndexedDB for data.

### Sync Triggers and Background Sync

```
APP LIFECYCLE EVENTS                      SYNC ACTION
─────────────────────────────────────────────────────
App launched (cold start)         →  Full sync if > 30 min since last
App foreground (from background)  →  Full sync if > 15 min since last
User pulls to refresh             →  Forced full sync
User submits a review             →  Partial sync (reviews only)
App goes to background            →  Flush pending writes
Background fetch (iOS/Android)    →  Availability + reviews only
```

**Background sync notes:**
- iOS restricts background fetch significantly. We register via `expo-background-fetch` but cannot guarantee execution. Treat as best-effort.
- Android is more permissive. Register a WorkManager job (via Expo) for 30-minute intervals.
- Never perform tile downloads in the background — too battery-intensive.

### Conflict Resolution (detailed)

| Conflict Type | Resolution Strategy | Rationale |
|--------------|---------------------|-----------|
| Same service metadata, local vs remote | Remote wins | Server is source of truth for city data |
| Availability record: local estimate vs remote actual | Remote wins | Actual beats estimated |
| User review: submitted offline, server now has newer | Both kept | Reviews are immutable append-only |
| Duplicate review (same fingerprint + service + < 1h) | Keep latest timestamp | User may have retried after connectivity failure |
| Deleted service (removed from Toronto Open Data) | Mark `is_active = false`, keep local record | Never delete — user may be navigating to it |

### Web: Leaflet (Recommended over Mapbox)

**Why Leaflet over Mapbox GL JS:**

| Criterion | Leaflet | Mapbox GL JS |
|-----------|---------|-------------|
| Cost | Free, no API key | Free tier limited, API key required |
| Bundle size | ~40KB gzipped | ~280KB gzipped |
| Tile flexibility | Any tile provider (OSM default) | Mapbox tiles (or self-hosted) |
| React integration | `react-leaflet` (mature, well-documented) | `react-map-gl` (Mapbox-specific) |
| Offline support | Via Service Worker + Cache API | Requires Mapbox-specific offline packs |
| Expo Web compatibility | Works as a web component | Works as a web component |
| Clustering | `leaflet.markercluster` plugin | Built-in with GL layers |
| Hackathon speed | Faster to set up | More complex setup |

**Verdict:** Leaflet wins for a hackathon. The API key requirement alone disqualifies Mapbox for a quick demo — getting a key approved, setting up billing safeguards, and configuring Mapbox styles takes time we don't have.

All read endpoints use Supabase's auto-generated PostgREST API. Write endpoints use Edge Functions for validation.

### Rate Limiting and Caching

**Rate limiting:**
- Supabase has built-in rate limiting at the project level
- Add a custom header check in Edge Functions to reject clients without a valid app version header
- Review submissions: rate limit at 1 per service per device fingerprint per hour

**Caching:**
- Enable HTTP caching headers on read endpoints: `Cache-Control: public, max-age=300` (5 minutes)
- Cloudflare (via Supabase's CDN) will cache these automatically
- The sync-delta endpoint should NOT be cached (varies by `since` parameter)

## G. SMS Interaction Flow

### Overview

A user without data access texts a simple keyword to a dedicated number. The system responds with the most relevant nearby service information in plain text.

**Phone number:** Provision a Twilio number (or use a shared short code). Example: `(416) 555-0123` or short code `54321`.

### Complete Flow

```
┌─────────────────────────────────────────────────────────┐
│                     USER'S PHONE                         │
│                                                         │
│  User texts: "SHELTER M5T"                              │
│  (keyword + postal code prefix)                         │
└──────────────────────┬──────────────────────────────────┘
                       │ SMS
                       ▼
┌─────────────────────────────────────────────────────────┐
│                      TWILIO                              │
│                                                         │
│  Receives inbound SMS                                   │
│  Forwards via HTTP POST webhook to:                     │
│  https://{project}.supabase.co/functions/v1/handle-sms │
└──────────────────────┬──────────────────────────────────┘
                       │ HTTP POST (webhook)
                       ▼
┌─────────────────────────────────────────────────────────┐
│              SUPABASE EDGE FUNCTION: handle-sms          │
│                                                         │
│  1. Parse Twilio POST body:                             │
│     - From: "+14165550001"                              │
│     - Body: "SHELTER M5T"                               │
│                                                         │
│  2. Normalize message body:                             │
```

## I. Open Questions for the Team

These questions have a material impact on architecture or UX. Answer them early.

**1. Google Maps API Key for Android**
`react-native-maps` on Android requires a Google Maps API key. Do we have a key, or should we use Apple MapKit (iOS only) for the demo and Leaflet for both web and Android? The alternative is to demo on iOS + web only, which avoids the billing/key setup entirely.

**2. Twilio / SMS Number Provisioning**
Getting a Twilio number takes ~15 minutes and requires a credit card (but is free for demos). Who owns the account? Do we want to demo SMS live or just show the architecture? If live demo: provision the number in the first 2 hours of the hackathon.

**3. Toronto Open Data API — Field Names**
The shelter occupancy dataset has changed field names between versions. We need one person to make a live API call early and confirm the actual field names. Don't assume the documentation is current. Budget 2 hours for data normalization.

**4. "No Account" Policy — How Strict?**
We've designed for fully anonymous use. But if we allow optional accounts (even just an email + magic link), we unlock better review quality, personalized alerts, and saved favorites. Is that in scope? The decision affects the entire auth architecture. Recommend: anonymous-only for MVP, optional account is V2.

**5. Offline-First on Web — Is It Worth It for the Demo?**
Implementing Service Workers for offline web support adds 4–6 hours of work. For the hackathon demo, we'll almost certainly have internet. Should we defer full web offline support and focus on mobile offline instead? Recommend: offline-first on mobile (expo-sqlite), best-effort on web (just show stale data with an indicator).

**6. What Does "Success" Look Like for the Demo?**
Is the goal to impress judges with technical depth (offline sync, SMS) or to show a polished, user-friendly interface for the end user? These require different time allocations. The SMS flow is technically impressive but takes time. The map + filter + offline experience is more immediately legible to non-technical judges. Align on this before splitting work.

**7. Data Seeding — Do We Use Live Toronto Open Data or a Fixed Seed?**
For a hackathon demo, a fixed seed (30–50 realistic Toronto service records) is more reliable than a live API that might be slow or return unexpected data. But live data is more impressive. Recommend: seed a fixed dataset as fallback, also wire up live ingestion, show both working.

**8. Accessibility Scope**
The target user population includes people with low literacy, cognitive differences, and varying technical ability. How deeply do we design for this? At minimum: large tap targets (>44px), high contrast (WCAG AA), simple language. Are there specific accessibility requirements the hackathon evaluators will assess?
